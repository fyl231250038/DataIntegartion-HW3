以下是为你们量身定制的 **3大特种小组** 分工方案：

---

### 🛡️ 第1组：核心枢纽组 (集成服务器 & XML引擎)
**定位：** 项目的大脑。负责搭建系统中间件，制定“世界语（标准XML）”和消息转发。
**相关性分析：** 集成服务器的唯一工作就是接收XML、转换XML、发送XML。因此，写服务器核心逻辑的人和写XML转换脚本的人必须是“穿一条裤子”的，两人高度绑定。

*   **队员 1A（后端主程）：**
    *   搭建集成服务器（Spring Boot）。
    *   开发跨系选课、退课的路由转发接口（接收请求 -> 调用 1B 的解析代码 -> 转发给目标院系）。
    *   协调小组间的接口规范制定（比如决定用HTTP POST传字符串）。
*   **队员 1B（XML与数据标准专家）：**
    *   编写所有的 XSD 验证文件（全局格式和A/B/C本地格式）。
    *   编写所有的 XSLT 转换脚本（实现异构格式到统一格式的相互转换）。
    *   编写 Java 工具类（利用 DOM4J/SAX 执行生成、解析、验证），直接提供给 1A 调用。

### 🏠 第2组：院系业务线组 (A/B/C子系统全栈)
**定位：** 项目的四肢。负责落地具体的教务管理系统，因为A、B、C三个系统在业务逻辑上基本一致（都是登录、看课、选课），完全可以由两个人统包。
**相关性分析：** 前端页面和本地后端接口必须严密对接。前端画一个按钮，后端给一个API，两人联调频率极高，最适合结对开发。

*   **队员 2A（院系后端开发）：**
    *   编写 A、B、C 三个系统的本地后端逻辑（一套代码，换三个数据库配置即可）。
    *   实现本地教务的核心功能：登录校验、查询本地数据库的课程/学生。
    *   对接第1组：当本地学生选跨系课程时，调用 API 将请求打包发给集成服务器；同时开放接口，接收集成服务器发来的外部选课请求并写入本地库。
*   **队员 2B（GUI与前端开发）：**
    *   开发 A、B、C 系统的可视化用户界面（Vue网页或Java桌面UI均可）。
    *   实现“本地选课”和“跨系选课”、“退课”的交互页面。
    *   开发集成服务器端的**“全局统计大屏”**（展示总学生数、各学院选课热度等）。

### 🗄️ 第3组：数据基建与分析组 (DBA & QA与文档)
**定位：** 项目的基石和门面。向下夯实底层异构数据库，向上负责全局数据统计分析，同时为**“作业4（可视化）”**打基础，并产出能拿高分的报告。
**相关性分析：** 假数据怎么造、数据库怎么建，直接决定了全局统计好不好做、可视化好不好看，以及测试能不能测出Bug。因此数据层与测试/统计层天然绑定。

*   **队员 3A（异构数据工程师 DBA）：**
    *   负责部署 MySQL、SQL Server、Oracle 数据库（解决环境配置问题）。
    *   严格按照PDF编写建库建表SQL脚本。
    *   **关键任务：** 编写 Python (Faker) 或 SQL 脚本，批量生成 150个学生、30门课、750条选课的**Mock数据**。（注意：故意在数据中制造一些“偏好”，比如A院系学生疯狂选B院系某门课，为作业4的图表展示做铺垫）。
*   **队员 3B（数据统计后端、QA与文档统筹）：**
    *   开发**全局统计功能 API**（供 2B 的大屏调用）。具体做法可以是让集成服务器定期去ABC拉取数据并聚合，或者直接连接ABC三个库做联合查询。
    *   担任测试工程师（QA）：模拟真实学生走通“选课-退课-跨系共享”的全链路，寻找Bug。
    *   **负责拿高分：** 绘制跨系选课/退课流程图、架构图，统筹另外两组的截图，撰写最终的大作业报告。

### 🚀 这种分工下的“联调工作流”是怎么样的？

这种分组最大的优势在于，**前期的开发大家可以关起门来在小组内完成，互不干扰：**

1.  **准备阶段：** 第3组（3A）建好数据库，把数据库连接IP（如果是云服务器）或SQL脚本发给第2组。
2.  **各自发育：**
    *   **第1组**内部联调：1A 塞一个假XML字符串给自己的代码，调用 1B 的转换工具，看看能不能正确输出另一个XML。
    *   **第2组**内部联调：2B 画好页面，点击按钮调用 2A 的接口，看看能不能从本地数据库查出课程展示出来。
    *   **第3组**发力：3B 开始研究怎么写全局聚合统计的SQL/代码，3A生成海量数据灌入数据库。
3.  **大联调（系统对接）：**
    *   第2组（A系统）点击“跨系选课” -> 发送请求给第1组（集成服务器）。
    *   第1组收到后，转换格式 -> 路由给第2组（B系统后端）。
    *   第2组（B系统）处理完成后返回成功 -> 第1组再原路返回给A系统。
    *   第3组（3B）在旁边看着，一旦哪里报错，立刻记录并协调双方修改；如果跑通了，立刻截图写进报告。
